<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| HTTP client → FastAPI request body | Untrusted JSON deserialized by Pydantic into AgentCreate / StatusChangeRequest |
| ORM column → SQLite | Persisted state crossing into storage; FK + CASCADE controls referential integrity |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Tampering | AgentCreate accepting `status` field | mitigate | Remove `status` field from AgentCreate entirely — Pydantic v2 with default `model_config` rejects unknown fields silently (extra="ignore"), so callers POSTing `{"status":"active"}` get an Agent with DRAFT status regardless of body. Service layer additionally hard-sets DRAFT in Plan 02 as defense in depth. |
| T-01-02 | Tampering | StatusChangeRequest.status accepting arbitrary strings | mitigate | Typed as `AgentStatus` enum (not `str`). Pydantic v2 enum validation auto-rejects unknown values with 422 — frontend cannot send `{"status": "deleted"}` to bypass state machine. |
| T-01-03 | Information Disclosure | AgentRead leaks internal enum repr in JSON | mitigate | `ConfigDict(use_enum_values=True)` serializes `AgentStatus.ACTIVE` as `"active"` not as Python repr — predictable contract for frontend. |
| T-01-04 | Repudiation | History row orphaned after Agent deletion | mitigate | `ForeignKey("agents.id", ondelete="CASCADE")` on AgentStatusHistory.agent_id + `cascade="all, delete-orphan"` on Agent.status_history. Deleting an Agent removes its history rows atomically. This is intentional for v1 (no requirement to retain history of deleted Agents); v2 may revisit. |
| T-01-05 | Denial of Service | AgentStatusHistory.reason unbounded growth | mitigate | `String(500)` cap on reason column — caller cannot insert 10MB reason strings. |
</threat_model>
